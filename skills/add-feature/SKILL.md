# Feature 实现工作流

把一个 Feature 从需求落到「经过客观审查的实现」。**核心门控**：实现完成不等于完成——必须通过隔离上下文的子代理审查。

## Step 1: 理解需求

- 入参是描述或 GitHub Issue 链接（`gh issue view <url>`）。
- 澄清：要解决什么问题、谁用、验收标准（≥2 条可验证条件）。模糊处**先问再做**。
- 定位归属项目（子模块）。读取该项目的档案（`project-profiles.md`）与既有约定。

## Step 2: 方案设计

- 先看现有架构与公共能力（主仓 `libs-*`、`contracts/` 契约），优先复用。
- 给出实现要点：涉及的层、接口/契约改动、数据/配置改动。
- 改动面大或涉及跨子模块契约时，先与用户对齐方案再编码。

## Step 3: 实现

- 按项目既有模式编码（命名、分层、错误处理、日志与既有代码一致）。
- 类型注解完整；新增业务逻辑**同步补测试**。
- **红线**：不得硬编码凭据/主机 IP/客户信息；示例数据用虚构占位。

## Step 4: 自检

实现者自检（lint / 编译 / 相关测试通过）。但**自检不能替代客观审查**——实现者对自己的代码满意度天然偏高。

## Step 5: 内嵌客观审查（硬门控）⭐

用 **Agent 工具拉起 `code-reviewer` 子代理**（隔离上下文、只读）审查本次变更。它会自取 diff、按 `code-review` rubric 逐维度审查、输出 `[B*]`/`[W*]` 与裁决。

- 裁决为 **REQUEST_CHANGES** 或存在未清零 `[B*]` → 用 `/fix-review` 处理，然后**再次拉起子代理复审**，直到 `[B*]` 清零。
- **APPROVE** → 进入收尾。
- **NEEDS_DISCUSSION** → 把讨论项呈给用户定夺。

> 为什么用子代理而非自审：同一会话里实现者已形成「这代码没问题」的锚定，难以发现自身盲区。隔离上下文 + 旁观者视角才能客观暴露问题。

## Step 6: 收尾

- 汇总：实现了什么、审查结论、测试结果。
- 询问用户是否提交/开 PR。提交信息遵循仓库约定，**不含客户/凭据信息**。
- 若源于 GitHub Issue，提示可在 PR 关联 `Closes #<n>`。
