# Issue 修复工作流

把一个 Bug 从「现象」修到「根因闭环且经过回归」。**核心原则**：先复现、修根因、再回归——不在没复现的情况下盲改。

## Step 1: 获取 Issue 上下文

- 入参是 GitHub Issue 链接（`gh issue view <url> --comments`）或 Bug 描述。
- 提取：症状、复现步骤、期望/实际结果、错误日志/堆栈、环境、可复现性。信息不足**先追问**，不臆测。
- 定位归属项目（子模块）。

## Step 2: 复现

- 按复现步骤在本地/对应环境复现。复现不了**不要直接改码**——转为收集更多证据或与报告人对齐。
- 用日志/trace/调试缩小范围，定位到具体文件:行号。

## Step 3: 根因分析

- 区分**根因 vs 表象**：错误信息往往是症状，沿调用链回溯到真正成因。
- 判断影响面：是否同一根因还会触发其他路径？是否跨子模块？

## Step 4: 修复

- **修根因，不打补丁**。若只能临时缓解，明确标注 TODO 并关联 Issue。
- 遵循项目既有模式；复用公共能力。
- **红线**：不得在代码/日志/测试里引入凭据/真实 IP/客户信息。

## Step 5: 回归与防回归

- 加一条**能复现该 Bug 的测试**（先红后绿），确保问题不再回归。
- 跑相关测试套件（命令见 code-review 的 `resources/<project>.md`「验证」节）。

## Step 6: 内嵌客观审查

用 Agent 工具拉起 **`code-reviewer` 子代理**审查修复变更；按裁决用 `/fix-review` 处理 `[B*]` 直到清零（同 add-feature Step 5）。

## Step 7: 收尾与关单

- 汇总：根因、修复点、回归结果。
- 询问是否提交/开 PR；PR 描述关联 `Closes #<n>`，合并后自动关单。
- 如需直接更新 Issue：`gh issue comment <url> --body-file ...` 说明根因与修复（正文先用 Write 写临时文件，避免 shell 解析中文/代码块）。
