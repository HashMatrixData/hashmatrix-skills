# 架构级代码审查

从项目长期可维护性角度审核代码变更。**核心判断**：变更是从架构出发的方案，还是绕过架构的补丁？

> **本 Skill 是审查 rubric 的唯一源**。`code-reviewer` 子代理（`agents/code-reviewer.md`）通过 `skills:` 预加载本文件复用同一套 rubric；`add-feature`/`fix-issue` 在实现/修复后内嵌拉起该子代理，在**隔离上下文**做客观审查并以「❌ 未清零不放行」硬门控。手动审查任意范围/PR 时直接用本 Skill。

> **项目差异**：本文件只放所有项目通用的审查维度。各子模块的架构/类型/测试特化见 `{baseDir}/resources/<project>.md`（按 `_template.md` 格式，随各项目技术选型落地后补充；缺失时按本主流程执行）。

## Step 0: 解析输入参数

本 Skill 接受两个参数，均可为空：**审查范围**（第一个参数）与**审查重点**（第二个参数）。

| 输入示例 | 范围 | 重点 |
|---------|------|------|
| （空） | 工作区 | 无 |
| `abc1234` | 该 commit | 无 |
| `abc..def` | commit 范围 | 无 |
| `--pr` | 当前分支 vs main | 无 |
| `https://github.com/.../pull/123` | GitHub PR | 无 |
| `abc1234 重点看错误处理` | 该 commit | 错误处理 |
| `重点看权限校验` | 工作区 | 权限校验 |

**拆分逻辑**：尝试把第一个 token 匹配为范围标识（commit hash `/^[0-9a-f]{6,40}$/`、含 `..` 的范围、`--pr`、含 `github.com` 的 URL）。匹配成功→首 token 为范围、其余为重点；匹配失败→工作区模式、整串为重点。

### 审查范围获取

| 范围 | 获取方式 |
|------|---------|
| 工作区（默认） | `git diff && git diff --cached` |
| commit hash | `git show <hash> --stat && git show <hash>` |
| `a..b` 范围 | `git log --oneline <range> && git diff <range>` |
| `--pr` | `git log --oneline main..HEAD && git diff main...HEAD` |
| GitHub PR URL | `gh pr view <url>` + `gh pr diff <url>`（或 `mcp__plugin_github_github__pull_request_read`） |

**通用规则**：先 `--stat` 了解范围，再读完整 diff。变更文件 > 15 个时展示列表让用户选择重点。审查重点（若有）在 Step 2 每个维度中**叠加**为额外关注方向。

## Step 1: 理解变更上下文

对每个变更文件用 Read 读取**完整文件**（不只是 diff），理解：变更所在模块的职责与架构层次、函数/类在调用链中的角色、相关接口/基类。

**按项目类型**：参见 `{baseDir}/resources/<project>.md` 的「架构上下文」章节。

## Step 2: 逐维度审查

每个维度给出 **✅ 通过 / ⚠️ 建议改进 / ❌ 需要修改** 三级评价。

### 2.1 架构一致性
- **补丁 vs 方案**：变更是符合架构的有机扩展，还是绕开架构的临时补丁？
- **分层边界**：是否存在跨层调用/职责越界？
- **DRY / 复用**：是否创建了与已有代码重复的逻辑？是否有现成工具/公共库可复用？
- **依赖方向**：是否出现底层反向依赖上层？
- **配置管理**：新增配置是否走正规渠道，而非硬编码？

### 2.2 类型安全
- 函数签名是否有完整类型注解/TypeHint？泛型是否具体化（`list[str]` 而非裸 `list`，`Array<T>` 而非 `any[]`）？
- 是否有不安全的类型断言/转换？缺失类型注解是否有合理理由（需用户确认后才可放行）？

### 2.3 测试质量
- 核心业务逻辑路径是否有测试覆盖（含 error path 和边界条件）？
- 测试是否在正确层级（单元/集成/E2E）？是否与项目已有约定一致？
- 无测试的业务逻辑变更标记为 ⚠️。

### 2.4 错误处理与可观测性
- 异常是否被正确捕获和传播？有无吞掉异常的 bare `except:`/空 `catch`？
- 面向用户的错误信息是否有意义、是否泄露内部细节？
- 关键操作是否有日志/trace span？日志级别是否恰当？

### 2.5 安全性
- 认证/授权：新增 API 是否配置了正确的认证/鉴权？
- 输入校验：用户输入是否经过验证？是否存在注入风险？
- **敏感数据（红线）**：代码/日志/配置中是否泄露密码/token/API key/真实主机 IP/客户可识别信息？命中即 `[SEC]` 阻塞。
- 多租户隔离（如适用）：查询是否正确限定租户范围？

### 2.6 代码质量
- 函数复杂度（过长需关注）、命名语义、临时代码（TODO/HACK 是否关联 issue）、未清理的调试代码。

### 2.7 条件维度（按变更内容触发）
**按项目类型**：参见 `{baseDir}/resources/<project>.md` 的「条件维度」章节（如数据迁移安全、并发与资源、Operator reconcile 等）。

## Step 3: 输出审查报告

### 问题类别标签

| 标签 | 含义 |
|------|------|
| `[PATCH]` | 补丁式修复 / 未解决根因 |
| `[DUPLICATE]` | 重复抽象 / 既有工具未复用 |
| `[LAYER]` | 分层架构违规 |
| `[TYPE]` | 类型安全问题 |
| `[TEST]` | 测试覆盖不足 |
| `[SEC]` | 安全问题 / 凭据泄露 |
| `[CONTRACT]` | API 契约违规 |
| `[PERF]` | 性能问题 |
| `[CONV]` | 命名/结构/规范问题 |

### 报告模板

```markdown
## 代码审查报告

### 变更概要
- 范围：<commit/工作区/PR>
- 涉及文件：N 个
- 变更类型：<新功能/Bug修复/重构/测试/文档>

### 审查结果
| 维度 | 评级 | 说明 |
|------|------|------|
| 架构一致性 | ✅ / ⚠️ / ❌ | 一句话概述 |
| 类型安全 | ... | ... |
| 测试质量 | ... | ... |
| 错误处理 | ... | ... |
| 安全性 | ... | ... |
| 代码质量 | ... | ... |

### ❌ 必须修改（阻塞合并）
#### [B1] [标签] 问题标题
- **文件**: `path/to/file:行号`
- **问题**: 描述
- **建议**: 修复方案（含代码示例）

### ⚠️ 建议改进
#### [W1] [标签] 问题标题
- **文件**: `path/to/file:行号`
- **问题**: 描述
- **建议**: 修复方案

### 💡 讨论项
架构层面的思考和设计决策讨论

### 🟢 亮点
值得肯定的好实践

### 总结
变更整体质量评价 + 是否建议合并
**[APPROVE / REQUEST_CHANGES / NEEDS_DISCUSSION]**
```

> **PR 模式**：报告输出后询问用户是否回写为 PR 评论（GitHub: `gh pr comment <url> --body-file ...` 或 `mcp__plugin_github_github__add_comment_to_pending_review`）。

## 审查原则

不要只看 diff（读完整文件）| 区分补丁与方案（补丁标 ❌ 给替代）| 区分严重等级 | 给具体修复方案 | 尊重现有模式 | 肯定优点
