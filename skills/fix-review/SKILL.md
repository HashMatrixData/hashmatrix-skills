# 修复 Code Review 问题

消费 `code-review` / `code-reviewer` 子代理的审查报告，把 ❌ 阻塞项清零、按需处理 ⚠️ 建议项。**审查与修复职责分离**：审查只读、修复只在主上下文做。

## Step 1: 获取并解析报告

来源优先级：参数显式传入 > 上下文最近一次审查报告 > 询问用户。

解析出两类条目（依赖 code-review 的编号约定）：
- **`[B*]` 阻塞项**：必须修复或明确豁免后才能放行。
- **`[W*]` 建议项**：默认询问用户是否一并处理。

每条提取：编号、标签（`[SEC]`/`[LAYER]`/…）、文件:行号、问题、建议。

## Step 2: 逐项验证真实性（关键，过滤误判）

**不要无脑照搬审查结论。** 对每个 `[B*]`：用 Read 打开文件:行号确认问题确实存在、且建议方案成立。三种结果：

| 结论 | 处理 |
|------|------|
| ✅ 确认成立 | 进入 Step 3 修复 |
| ❌ 审查误判 | 标注「不修复 + 理由」，在汇总中说明（审查子代理客观但可能多报） |
| 🤔 需取舍 | 进入讨论模式，列出方案与权衡，请用户定夺 |

`[SEC]` 类（凭据/IP/客户信息泄露）一经确认**最高优先级**立即修复。

## Step 3: 分级修复

1. 先修 `[B*]`，按文件聚合修改，减少来回。
2. 每处修复遵循**根因优先**：审查标 `[PATCH]` 的，要给出架构层修复而非再打补丁。
3. 复用主仓公共能力（`libs-java`/`libs-ts`/`contracts`），不重复造轮子。
4. `[W*]` 建议项：默认列出让用户勾选；用户要求「全修」则一并处理。

## Step 4: 回归验证

- 跑该项目的 lint / 测试（命令见 code-review 对应 `resources/<project>.md` 的「验证」节）。
- 对修复涉及的逻辑补/改测试。
- 修复后**可再次拉起 `code-reviewer` 子代理复审**，确认 `[B*]` 清零。

## Step 5: 汇总

输出：已修复项（编号→改动摘要）、不修复项（编号→理由）、讨论项结论、回归结果。结尾给出是否已满足合并门控（所有 `[B*]` 已闭环）。

## 讨论模式

当条目涉及架构取舍、或与现有设计冲突时，不直接改码：列出 ≥2 个方案 + 各自代价 + 推荐，请用户拍板后再执行。
